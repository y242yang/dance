# Studio Hop 2.0 — App Store submission notes

Milestone reached 2026-07-10. Everything below reflects what shipped in this version and
what was done to get it submission-ready.

## What's new in 2.0

- **Accounts** via Sign in with Apple (only login method — no email/password).
- **Social layer**: usernames, follow/follow-request/accept flow, Followers/Following
  lists, Find Friends (username search, excludes yourself from results).
- **Saved classes and class log moved off-device**: previously local-only
  (`UserDefaults`); now live entirely in Supabase (`saved_classes`, `log_entries`),
  visible to the owner and their *accepted* followers only. No local/cloud dual-storage
  — Supabase is the single source of truth once signed in.
- **Profile picture** upload (Supabase Storage, `avatars` bucket, path
  `{userId}/avatar.jpg`, always overwritten in place — no orphaned old versions).
- **Mutual-class indicator**: viewing a follower's profile shows a small badge on any
  saved/logged class you *also* have saved/logged (`LogEntry.isSameSession`, matches by
  source class id, or for manual entries, same minute + instructor first name).
- **Committed Classes** section (renamed from "Class Log"): swipe left on a saved class
  to Commit (log it) or Delete (unsave); these are independent — committing doesn't
  unsave, deleting a log entry doesn't unsave the source class.
- **Class cancellation handling**: if a studio cancels a class you'd committed to, the
  log entry survives (detached, `is_canceled` flag set, shown with a "Canceled" badge)
  instead of silently disappearing — see `mark_log_entries_canceled()` trigger in
  `sql/social_schema.sql`.
- **Account deletion** (Profile → Delete Account) — required by Apple Guideline
  5.1.1(v). Implemented as a Supabase Edge Function (`supabase/functions/delete-account`)
  since deleting `auth.users` requires the service-role key, which can't ship in the
  client. Verified working end-to-end via direct `curl` test (sign in as a throwaway
  user, call the function, confirmed the account and its cascaded data are gone).

## Decisions made (for future reference)

- **Report/block mechanism**: deliberately **not** added for this submission. Reasoning:
  Guideline 1.2 (User-Generated Content) is enforced most aggressively on apps with
  broad/public UGC exposure — public feeds, comments, DMs. This app's only free text (a
  log entry's notes field) is visible exclusively to followers you've explicitly
  accepted, and there's no messaging feature. Judged as low-risk given that scope. If a
  future submission gets flagged for this, the cheapest fix is a "Report User" button
  that opens a pre-filled `mailto:` to the support email already published in the
  Privacy Policy — that alone satisfies the guideline without building real block
  relationship logic (hiding from search, preventing re-follow, etc.).
- **`display_name` column**: exists in `profiles` but is never populated or shown
  anywhere — Sign in with Apple's `.fullName` scope is requested but the callback never
  reads it out. Not a bug, just unused. Revisit if a "real name" feature is ever wanted.

## Known follow-ups (not blockers, just noted)

- No `supabase` CLI installed on this machine — the Edge Function was deployed via the
  Supabase Dashboard's in-browser editor instead. That editor has a known intermittent
  "entrypoint path does not exist" bug on deploy; a plain retry usually clears it.
- No unit tests exist yet for `SavedStore`'s pruning/cascade logic (only
  `LogEntryTests` for `LogEntry.isManual`/`isSameSession` — see `ios/DanceTests/`).
